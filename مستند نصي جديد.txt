## 📖 Introduction
CoreHub is a robust, high-concurrency real-time messaging system built on a Centralized Client-Server architecture. Designed with performance and reliability in mind, the primary goal of this framework is to provide a low-latency communication hub capable of managing thousands of concurrent users with a minimal memory footprint. It mimics the core mechanics of industry-leading chat applications, providing a seamless user experience.

---

## ✨ Key Features & Capabilities

### Phase 1: Security & Identity Management ✅
- **Secure Authentication:** Robust user registration and login endpoints.
- **Data Protection:** Passwords are mathematically hashed using `bcrypt` before reaching the database.
- **Session Management:** Utilizes **JWT (JSON Web Tokens)** to securely authenticate users during the WebSocket handshake.

### Phase 2: Advanced Real-Time UX ✅
- **Live Presence Indicators:** Dynamic tracking of `Online/Offline` user statuses. The server broadcasts presence updates instantly as TCP connections are opened or closed.
- **Ephemeral Typing Indicators:** Silent WebSocket events broadcasting *"User is typing..."* states. These events feature auto-hide UI timers and bypass the database entirely to ensure zero I/O overhead.
- **Delivery Acknowledgments (✓/✓✓):** Full message lifecycle tracking:
  - **🕒 Pending:** Message generated by the client (Offline/Queued).
  - **✓ Server Ack:** Acknowledged and persisted in PostgreSQL.
  - **✓✓ Delivery Ack:** Acknowledged by the recipient's active WebSocket client.
- **State Persistence:** Database integration utilizing `client_msg_id` and `status` columns to reliably retain message delivery states across sessions and browser refreshes.

### Phase 3: Scaling, Resiliency & Group Chats ✅
- **Group Chats (Rooms):** Users can create or join isolated group channels. The Go server efficiently routes and broadcasts messages only to active group members using synchronized memory maps.
- **Heartbeat Mechanism (Ping/Pong):** Advanced connection health tracking. The server enforces a 60-second read deadline, while the client sends silent pings every 30 seconds to actively detect network drops and prevent memory leaks.
- **Offline Queuing & Auto-Reconnect:** Graceful degradation during network failures. The client queues un-sent messages locally (marked as 🕒), silently attempts to reconnect every 3 seconds, and automatically flushes the queue upon reconnection.
- **Smart Delivery Synchronization:** Retroactive delivery acknowledgments. When a user logs in and fetches history, the client automatically sends `delivery_ack` signals for newly fetched unread messages to update the sender's UI to `✓✓` in real-time.

### Phase 4: Core Backend Enhancements ⏳ *(Next Steps)*
- **Database Timestamps:** Adding explicit `created_at` records for accurate message chronological sorting.
- **Data Pagination:** Implementing SQL Limit & Offset to fetch chat history in batches, optimizing memory usage for large conversations.
- **UI/UX Polishing:** Displaying message timestamps and "Last message" previews in the sidebar.

## 🏗️ Technical Architecture & Stack

The system is engineered using **Go (Golang)** to leverage its native concurrency features (Goroutines and Channels). This ensures safe and lightning-fast message routing across the server without race conditions.

* **Backend Language:** Go (Golang)
* **Real-time Protocol:** WebSockets (via `Gorilla WebSocket`) for full-duplex, persistent TCP connections.
* **Database & Persistence:** PostgreSQL for relational data storage and history archiving.
* **Frontend:** Vanilla HTML/CSS/JS for a lightweight, dependency-free, and responsive UI.

## 🗄️ Database Schema

To achieve data persistence, isolated group contexts, and message state tracking, the system relies on the following relational structure in PostgreSQL:

```sql
-- Users Table
CREATE TABLE users (
    username TEXT PRIMARY KEY,
    password TEXT NOT NULL
);

-- Groups Table
CREATE TABLE groups (
    id SERIAL PRIMARY KEY,
    name TEXT UNIQUE NOT NULL
);

-- Group Members Mapping Table
CREATE TABLE group_members (
    group_id INT REFERENCES groups(id),
    username TEXT REFERENCES users(username),
    PRIMARY KEY (group_id, username)
);

-- Messages Table (Updated for Group Routing)
CREATE TABLE messages (
    id SERIAL PRIMARY KEY,
    username TEXT NOT NULL,
    receiver_username TEXT, -- Nullable/Empty for group messages
    group_id INT DEFAULT 0, -- 0 for private messages, >0 for groups
    content TEXT NOT NULL,
    client_msg_id TEXT,
    status TEXT DEFAULT 'sent'
);
